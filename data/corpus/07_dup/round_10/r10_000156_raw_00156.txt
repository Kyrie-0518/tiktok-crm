## 三、服务器部署架构

```
互联网 HTTPS 443
    ↓
宿主机 Nginx (/etc/nginx/conf.d/bozone.conf)
    ├── /api/*      → 127.0.0.1:3000  (bozone-server, network_mode: host)
    ├── /uploads/*  → 127.0.0.1:3000
    └── /*          → 127.0.0.1:8080  (bozone-client 容器)
```

| 容器 | 服务 | 技术 |

## 六、欧文智能体（核心 AI）

### 运行流程

```
用户输入 → agentLoop(channels, query)
    → 第1轮: AI + Tool定义 → 识别需要调用哪些工具
    → 执行工具（数据库查询）
    → 第2轮: AI + 工具结果 → 再决定是否需要更多数据
    → ...（最多5轮 Tool Calling）
    → 最终AI生成报告 → 返回 report 字符串
```

### 关键文件

| 文件 | 作用 |
|---|---|
| `server/src/routes/agent-chat.ts` | 核心逻辑：`agentLoop` + 系统提示词 + 工具定义 + 工具执行 |
| `server/src/routes/ai.ts` | AI 通道管理（DeepSeek/火山引擎/OpenAI） |
| `server/src/routes/ai-channels.ts` | 通道配置 CRUD |

### 可用工具（Function Calling）

| 工具名 | 功能 |
|---|---|
| `get_shop_stats` | 店铺销售概览（GMV/订单/SKU/趋势） |

## 七、飞书 Bot（最新状态）

### 架构

```
飞书用户发消息
    ↓
飞书服务器 → WebSocket 长连接
    ↓
bozone-server（bot-feishu-ws.ts）
    ├── Lark.WSClient 接收事件
    ├── handleMessage() 处理消息
    ├── agentLoop() 调用AI
    └── sendMessage() 回复（调用飞书 API）
```

### 关键配置

| 飞书后台项 | 值 |
|---|---|
| 订阅方式 | 使用长连接接收事件 |
| 事件 | `im.message.receive_v1`（接收消息 v1） |
| 权限 | `im:message`, `im:message:send_as_bot` |
| 应用类型 | 企业自建应用 |

### 为什么用长连接而不是 Webhook

飞书 Webhook 需要在后台保存回调 URL，保存时飞书会向该 URL 发送 POST 挑战码验证。但阿里云 ECS 对飞书侧 IP 有网络拦截 → Nginx access.log 无任何飞书请求 → 无法通过验证 → 最终采用长连接方案（服务端主动连飞书 WebSocket）。

### 已解决的 Bug

| Bug | 原因 | 修复 |
|---|---|---|
| 无 `send_as_bot` 权限 → 消息发不出去 | 飞书权限未配置 | 添加 `im:message:send_as_bot` |
| 同一条消息回复多次 | 飞书 3 秒超时重推事件 | 加 `processedMessages` Set 去重 |
| 飞书加密解密错误 | Key=base64(key) 应为 SHA256 | 按文档修正为 SHA256 + IV 从密文提取 |
